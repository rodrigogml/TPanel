**Status**: Aprovado

## 1. Visao e Proposito

**O que e**: TPanel, abreviacao de Turin Panel, e um portal web para monitoramento e administracao de servidores Linux, inicialmente Debian 13, executado em PHP sobre Apache. [fornecido]

**Problema que resolve**: Centraliza a leitura da saude do servidor e a execucao segura de acoes administrativas, sem permitir comandos arbitrarios. [fornecido]

**Proposta de valor**: Entregar um painel simples, seguro, rapido, modular, responsivo e com aspecto visual premium para operacao de servidores e servicos Linux. [fornecido]

## 2. Usuarios e Stakeholders

| Ator | Papel | Acoes Principais |
|------|-------|------------------|
| Administrador | Usuario administrativo | Monitora o servidor, consulta logs, visualiza alertas, reinicia servicos de forma controlada e executa acoes administrativas autorizadas. |
| Monitor | Usuario de acompanhamento | Visualiza dashboards, metricas, logs e alertas sem executar acoes administrativas criticas. |

**Stakeholders de decisao**: O proprio mantenedor do projeto, com desenvolvimento inicial por 1 dev. [fornecido]

## 3. Escopo

### MVP (Essencial)

1. Dashboard geral de saude do servidor com status geral, CPU, memoria, disco, RAID, rede, Docker, alertas e uptime. [fornecido]
2. Paginas de monitoramento de sistema, CPU, memoria, armazenamento, rede, processos, logs, seguranca e uptime. [fornecido]
3. Visualizacao e administracao controlada de servicos systemd, Docker e containers. [fornecido]
4. Logs recentes, journal, syslog, erros recentes e alertas operacionais. [fornecido]
5. Auditoria completa de todas as acoes administrativas. [fornecido]
6. Executor seguro de comandos previamente autorizados, com validacao de parametros e timeouts. [fornecido]
7. Monitoramento completo de SMART, saude dos discos, RAID, sensores, temperaturas, ventoinhas e energia quando disponivel. [fornecido]
8. Visualizacao e gestao de agendamentos via cron e timers systemd. [fornecido]
9. Layout responsivo com menu lateral recolhivel, barra superior, pesquisa, usuario, alertas, tema claro/escuro e adaptacao para celular. [fornecido]
10. Configuracoes sensiveis fora do Git, com arquivos reais ignorados e modelos versionados com extensao `.model`. [fornecido]

### Pos-MVP (Desejavel)

1. Graficos historicos mais avancados. [fornecido]
2. Retencao de metricas com politica configuravel. [fornecido]
3. Tela auto atualizavel por intervalos configuraveis para monitoramento em tempo real pelo painel. [fornecido]
4. Notificacoes usando o processo do projeto NotiCLI. [fornecido]
5. Administracao mais ampla de servicos, Docker e atualizacoes. [inferido]
6. Suporte futuro a novos modulos sem alterar a arquitetura. [fornecido]

### Fora de Escopo

- API REST nesta fase. [fornecido]
- Execucao de comandos arbitrarios. [fornecido]
- Dependencia obrigatoria de servicos pagos. [fornecido]
- Monitoramento centralizado de multiplos servidores no MVP. [inferido]

## 4. Prioridades e Trade-offs

**Ordem de prioridade**: Seguranca > Qualidade/robustez > Experiencia visual premium > Performance > Velocidade de entrega > Escopo completo. [fornecido]

**Decisoes explicitas**:

- Priorizar seguranca mesmo que reduza velocidade de entrega. [fornecido]
- Manter acoes administrativas restritas a comandos autorizados. [fornecido]
- Usar recursos locais/open source e evitar dependencias pagas no inicio. [fornecido]
- Preservar visual premium, CSS de qualidade e experiencia responsiva como requisito do produto. [fornecido]

## 5. Restricoes

| Restricao | Valor | Notas |
|-----------|-------|-------|
| Prazo | Sem prazo fixo | Projeto evolutivo. [fornecido] |
| Equipe | 1 dev | Desenvolvimento inicial por um desenvolvedor. [fornecido] |
| Budget | Recursos locais/open source | Sem dependencia de servicos pagos nesta fase. [fornecido] |
| Tecnica | PHP, Apache, Debian 13, MySQL | Aplicacao monolitica em PHP sobre Apache, inicialmente para Debian 13, com instancia MySQL. [fornecido] |
| Seguranca operacional | Apache autentica, PHP usa a identidade autenticada, `www-data` executa apenas comandos autorizados via sudo | Criar usuario/grupo Linux `tpanel`; scripts e executaveis pertencem a `tpanel`. [fornecido] |
| GitHub | Repositorio remoto `git@github.com:rodrigogml/TPanel.git` | Projeto preparado para publicacao no GitHub. [fornecido] |

## 6. Stack Tecnica

| Camada | Tecnologia | Justificativa |
|--------|------------|---------------|
| Backend | PHP monolitico | Stack definida para execucao sobre Apache, simples de hospedar e operar. [fornecido] |
| Frontend | HTML/CSS/JavaScript responsivo dentro da aplicacao PHP | Interface web responsiva sem API REST nesta fase. [inferido] |
| Banco de dados | MySQL | Instancia MySQL definida para persistencia de metricas, auditoria e configuracoes aplicaveis. [fornecido] |
| Infraestrutura | Apache em Linux Debian 13 | Ambiente inicial alvo do portal. [fornecido] |
| Execucao de comandos | Sudo restrito para comandos mapeados e autorizados | Impede execucao arbitraria e permite auditoria. [fornecido] |
| Scripts do sistema | Scripts Linux pertencentes a usuario/grupo `tpanel` | Separa responsabilidades e reduz superficie de permissao. [fornecido] |
| Notificacoes | NotiCLI por chamada CLI nao interativa | O README do NotiCLI descreve `noticli send` com `sender`, `category`, `priority`, `title`, `message` e configuracao centralizada. [fornecido] |

## 7. Qualidade e Padroes

**Padroes adotados**:

- Seguranca primeiro: OWASP, sudo restrito, validacao de parametros, timeouts e auditoria completa. [fornecido]
- Documentacao forte: specs, modelos de configuracao `.model`, decisoes registradas e preparo para GitHub. [fornecido]
- Performance: painel leve, rapido e sem dependencias desnecessarias. [fornecido]
- Observabilidade: logs claros de acoes, erros, alertas e eventos relevantes. [fornecido]
- Testes essenciais: foco nos servicos criticos, validacao de comandos, permissoes e comportamento de seguranca. [fornecido]
- Responsividade e acessibilidade: suporte a desktop, tablet e celular, com tema claro/escuro. [fornecido]

**Compliance**: Nenhum compliance formal especifico definido nesta fase; seguranca operacional e auditoria sao requisitos centrais. [inferido]

## 8. Integracao NotiCLI

O TPanel deve usar o NotiCLI como processo externo de notificacao. [fornecido]

Referencia do projeto: `https://github.com/rodrigogml/NotiCLI`. [fornecido]

Referencia local lida: `/home/rodrigogml/projects/NotiCLI/README.md`. [fornecido]

Modelo inicial de integracao:

- Chamar `noticli send` de forma nao interativa para eventos relevantes do painel. [inferido]
- Usar `--sender` com identificador do TPanel, respeitando o limite de ate 20 caracteres descrito no README do NotiCLI. [inferido]
- Usar `--category` para separar tipos de evento, como `server`, `service`, `security`, `disk`, `raid`, `docker` ou `backup`. [inferido]
- Usar `--priority` para classificar alertas como `HIGH`, `NORMAL` ou `LOW`. [inferido]
- Enviar `--title` e `--message` com conteudo operacional sem segredos. [inferido]
- Manter os segredos e rotas no arquivo de configuracao do NotiCLI, evitando duplicar credenciais no TPanel. [inferido]
- Registrar no TPanel o resultado da chamada e o codigo de saida do NotiCLI para auditoria e troubleshooting. [inferido]

- Evoluir tela auto atualizavel por tempos configuraveis para monitoramento em tempo real pelo painel. [fornecido]

**12 meses**:

- Amadurecer historico, retencao, alertas e notificacoes. [inferido]
- Preparar o projeto para publicacao e colaboracao no GitHub. [inferido]

**Riscos conhecidos**:

- Permissoes sudo mal delimitadas podem comprometer o modelo de seguranca. [inferido]
- Coleta de metricas muito frequente pode afetar desempenho em servidores menores. [inferido]
- Retencao historica sem politica clara pode crescer demais no MySQL. [inferido]
- Integracao de notificacoes deve evitar vazamento de segredos em titulos, mensagens, logs ou diagnosticos. [inferido]
